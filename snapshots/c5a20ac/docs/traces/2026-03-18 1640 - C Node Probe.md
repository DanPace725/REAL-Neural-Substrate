## Node-level read

### `n3` is the clearest split

`task_b` on `n3`:

- route transforms: `rotate_left_1: 6`
- no latent promotion at `n3`
- contradiction eventually becomes high (`peak 0.88997`)
- after early routing, behavior shifts into maintenance/investment rather than repeated transform commitment

`task_c` on `n3`:

- route transforms: `rotate_left_1: 3`, `xor_mask_1010: 8`
- contradiction remains `0.0` throughout the late repeated route window
- late cycles `25-32` repeatedly choose `route_transform:n9:xor_mask_1010`

Critically, those late `task_c` route choices happen while the node has a packet and a task, but before latent context is promoted:

- cycle `25`: latent available `1.0`, confidence `0.28942`, effective confidence `0.0`, promotion-ready `0.0`
- cycle `30`: latent available `1.0`, confidence `0.47695`, effective confidence `0.47695`, promotion-ready `0.0`
- cycle `32`: latent available `1.0`, confidence `0.60893`, effective confidence `0.60893`, promotion-ready `0.0`

So `n3` is making hard transform commitments on `task_c` while still below promotion threshold.

### `n4` has a different failure mode

`task_c` on `n4` does get strong latent resolution:

- cycle `21`: latent confidence `1.0`, promotion-ready `1.0`, growth-ready `1.0`, action `route_transform:n10:xor_mask_1010`
- cycles `23-24`: latent confidence still high (`0.85905`, `0.66356`), still choosing `xor_mask_1010`

But it is not perfectly stable:

- cycle `22`: with promoted context (`0.84494`, promotion-ready `1.0`), it still takes `route_transform:n9:rotate_left_1`

So even after successful latent promotion, downstream route selection can still mix in a stray transform.

## Interpretation

The data suggests two different breakdowns:

1. `task_c` lets downstream nodes commit transform-specific routes before latent context is promoted.
2. Even after promotion, downstream selection can still leak into an alternative transform branch.

The first issue looks more important. `n3` is repeatedly routing with low-confidence latent state but no contradiction pressure, which means the system is not treating that behavior as obviously problematic. In practice this looks like a selector-policy issue, not a morphogenesis issue:

- route-transform actions remain available under ambiguous latent state
- downstream nodes can settle into a transform-specific habit before the context estimate is mature
- once that habit starts paying locally, contradiction does not rise enough to force correction

## Most justified next move

The strongest next intervention is to gate or sharply penalize non-identity `route_transform:*` actions when:

- latent context is available
- effective context is not yet promoted
- the packet has a task id but no visible context

That would specifically target the `n3 task_c` pattern observed here without disabling latent routing entirely.
