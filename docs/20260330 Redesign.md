# REAL redesign idea: graph geometry, pulse dynamics, and functional decomposition

## High-level diagnosis

REAL already has many of the right ideas:

* local agent-like nodes
* ATP budget
* multi-dimensional coherence
* substrate memory
* growth request and slow-layer adjudication
* slice-based regulation

But the current bottom-stack abstraction is likely too compressed. Too many functions are being forced through the same node/update/routing mechanism:

* signal propagation
* ambiguity handling
* plasticity/update permission
* structural stress / growth demand
* context retention
* regulator interaction

The result is that the architecture is both rich and overburdened.

The redesign goal is **not** “add more random complexity.”
The goal is:
**separate the overloaded functions at the right resolution, while preserving REAL’s local, interpretable, non-backprop character.**

---

## Part 1: graph idea

There are two related graph intuitions.

### 1. Overlap-growth / ternary graph intuition

Earlier graph experiments suggested that some local connection rules naturally create:

* regular planar lattice-like closure
* while other local rules generate recursive overlap patterns that seem to “want” a higher-dimensional or more folded structure

Core idea:

* some local rules are happy to tile or extend in 2D
* others create constraint pressure that requires richer embedding, recursive overlap, or nonlocal relation

This is relevant because REAL may be hitting the same issue architecturally:

* some tasks are solvable by extending the current routing substrate
* other tasks may require a different kind of internal organization, not just more nodes or more layers

### 2. Practical graph takeaway for REAL

Instead of treating each node as a simple scalar unit with generic edges, consider a **more structured local primitive**:

* typed ports
* role-differentiated connections
* possibly a small internal motif instead of a single abstract node

This does **not** mean implementing a literal hypergraph immediately.
It means using the graph intuition to justify:

* structured local connectivity
* role-specific channels
* and non-interchangeable relations

The practical coding interpretation is:

**Edges should not all mean the same thing.**
Potential edge/channel types:

* evidence / feedforward
* context / lateral
* regulatory / feedback
* maybe structural / growth-related

This is a functional decomposition move, not an aesthetic one.

### 3. Concrete bounded ternary-overlap geometry

The graph idea above is not only abstract. A specific candidate geometry is:

* each node has up to **3 forward children**
* successive layers expand by overlap rather than by full Cartesian branching
* the intended bounded layer counts are:
  * layer 0: `1`
  * layer 1: `3`
  * layer 2: `7`
  * layer 3: `15`

This is **not** the same as a simple `3^n` tree.

The important difference is:

* adjacent parents in one layer share children in the next layer
* so growth is overlapping and intersecting rather than fully independent
* this creates a recursive `1, 3, 7, 15, ...` style expansion instead of `1, 3, 9, 27, ...`

The motivating intuition is that adjacent local units should be able to:

* preserve some distinct downstream branches
* while also projecting into partially shared downstream differentiators

That gives a middle ground between:

* a narrow single-route chain
* and a fully exploding branching tree

This overlap is important because some tasks may need:

* independent local specialization
* plus shared downstream recombination or disambiguation

In other words, the next layer should not just be "more separate branches."
It should also contain **intersections between nearby branches**.

### 4. Why this geometry might matter

The current `source -> routes -> sink` style substrate may be sufficient for simpler tasks, but it may be too shallow a relational primitive for harder tasks.

The bounded ternary-overlap idea is meant to test whether the substrate benefits from:

* richer local branch interaction
* explicit downstream recombination points
* bounded structural depth without runaway graph size
* a topology where ambiguity can be separated by partially shared child structure instead of only by transform choice on flat edges

This is especially relevant if current difficulty is not only about local pulse behavior, but also about the **geometry of available differentiation paths**.

### 5. Important separation from the pulse pilot

This graph-geometry idea should be treated as a **separate experimental axis** from the current pulse/local-unit pilot.

They are related, but they are not the same intervention:

* the pulse pilot asks whether the current node/routing abstraction needs better local state separation and thresholded commitment
* the overlap-graph proposal asks whether the current topology itself is too weakly structured

So the clean framing is:

1. test whether pulse/local-unit redesign improves C/HR behavior on the current topology
2. if not enough, test whether a bounded ternary-overlap topology gives the system better differentiation capacity
3. only after that consider combining both changes

## Part 2: RNA analogy

This is not about literally simulating RNA.

The useful abstraction is:

RNA is interesting because it can both:

* **store sequence information**
* and **participate in doing things**

That suggests a useful design principle:

**A local unit may need a compact internal adaptive state that both remembers and acts.**

In REAL terms, this means a node should probably not just have:

* current activity
* and generic substrate support

It may need a more explicit compact local state that influences:

* how incoming information is interpreted
* whether local state is ambiguous vs committed
* whether plasticity is allowed
* whether growth is requested
* how recent local history affects present behavior

This internal state should be:

* small
* interpretable
* persistent enough to matter
* not just another giant hidden-state blob

Think of it as a **local adaptive code/state**, not literal biology.

## Part 3: proposed bottom-stack redesign direction

### Replace the current overly generic node with a more decomposed local unit

Not necessarily a full new architecture yet. Start with a simple structured primitive.

### Minimal conceptual split inside each local unit

Each unit should distinguish at least three functions:

1. **Signal state**

   * immediate evidence accumulation
   * local prediction tendency
   * actual propagated output when threshold crossed

2. **Context / ambiguity state**

   * tracks asymmetry, uncertainty, conflicting evidence, unresolved alternatives
   * should allow “not yet committed” to be a real local state

3. **Plasticity / adaptation state**

   * tracks whether updates should occur
   * whether local experience should become substrate memory
   * whether a growth request is warranted
   * whether current local regime is failing persistently

This can be implemented as one node with multiple internal fields, or as a tiny motif/microcircuit.
Do **not** overcomplicate the first version.

## Part 4: pulse / action-potential idea

Slices may already be a coarse version of pulse logic.

Current slice idea:

* fast layer runs briefly
* summary is emitted
* slow layer updates policy/budget
* next slice fires

Interpretation:
**The architecture already has macro-level pulses.**

Possible next step:
introduce **micro-level pulse dynamics** at the node or local-unit level.

### Why

Current routing may be too continuous and too eager.
This likely hurts ambiguity-heavy tasks like C-family because weak/conflicting evidence can leak everywhere and cause premature commitment.

### Proposed local pulse behavior

Each local unit should:

* accumulate evidence over time
* only “fire” or propagate when threshold conditions are met
* possibly enter cooldown/refractory after firing
* allow subthreshold ambiguous information to persist locally without immediate propagation

This is not a full spiking-neural-net conversion.
Start with a simple mechanism:

* accumulation variable
* threshold
* fire event
* cooldown/refractory
* maybe separate ambiguity reservoir

### Expected benefits

* reduce always-on routing
* reduce premature collapse under ambiguity
* allow local evidence to build before commitment
* make slice summaries reflect meaningful committed activity rather than continuous low-grade leakage

## Part 5: growth redesign

Current issue:
growth is partly handled by fast-layer request and partly by slow-layer initiation, but the current implementation may still blur:

* growth demand
* growth approval
* growth retention

### Intended architecture

Growth should be:

1. **requested bottom-up** by fast-layer local stress/insufficiency
2. **adjudicated top-down** by the slow layer between slices
3. **preserved structurally** across later policy changes without requiring permanent commitment to a “growth mode family”

### Growth signal should be more explicit

A local unit should be able to emit something like:

* unresolved conflict persists
* ambiguity remains high
* topology insufficient
* ATP/useful effort mismatch
* current structure cannot separate alternatives cleanly

Then the slow layer decides:

* approve growth
* deny growth
* delay growth
* alter budget first
* try another policy first

### Important

Do not let growth become a generic favorite top-down policy just because it often helps.
Growth should be warranted by local fast-layer conditions.

## Part 6: memory / continuity principle

One likely problem is not “no memory,” but “wrong memory partitioning across mode transitions.”

The redesign should assume that different things have different continuity requirements.

### Things that may need stronger persistence

* substrate/support reflecting learned structure
* context memory
* ambiguity traces
* coherence history
* growth justification history
* effects of successful structural differentiation

### Things that can remain transient

* slice-local regulator preferences
* temporary routing excitement
* some ephemeral exploration traces

The goal is:
**policy changes should not erase meaningful solved structure.**
This is especially important around growth and mode switching.

## Part 7: concrete first implementation target

Do **not** implement everything at once.

### First experiment

Create a REAL variant where each node/local unit has:

* `signal_accumulator`
* `ambiguity_or_context_state`
* `plasticity_or_growth_state`
* thresholded firing
* cooldown/refractory
* typed incoming/outgoing channels if feasible

Minimal behavior:

* accumulate local evidence
* if threshold crossed, emit signal
* if ambiguity remains high, hold state without immediate commitment
* if persistent local insufficiency is detected, emit growth request signal
* substrate update occurs only under clearer plasticity conditions, not every active moment

### Test on

* small C-family subset
* small HR-family subset

### Main questions

* does overcommitment reduce?
* does ambiguity retention improve?
* do growth requests become more meaningful?
* does the slow layer need fewer scaffolding heuristics?
* does pulse-based local gating improve slice-level regulation quality?

## Very short version

REAL likely already has many of the right principles, but the base node is overburdened. Use the ternary/overlap graph intuition to justify structured local connectivity and typed channels, and use the RNA analogy to justify a compact local adaptive state that both remembers and conditions behavior. Add explicit separation between signal, ambiguity/context, and plasticity/growth states. Introduce thresholded node-level pulse dynamics so information can accumulate before firing. Make growth request originate bottom-up from fast-layer local stress and be adjudicated by the slow layer between slices. Start with a minimal node redesign, not a full rewrite, and test whether this reduces premature commitment and improves ambiguity handling on C-family and HR-family tasks.
